1. The problem we are solving

The problem is not that AI forgets conversations. Modern AI can already use chat history and persistent memory. The deeper enterprise problem is that AI does not reliably preserve institutional meaning and operating posture over time.

Finance-domain scenario

Banking credit risk

A Chief Risk Officer says:

Regional commercial real estate defaults are increasing, so we are tightening our lending posture for mid-sized retail mall projects.

To a human banker, this is not just a sentence. It signals a change in the bank’s credit behavior.

What a human understands
Words usedInstitutional meaning
Defaults are increasingCredit risk is worsening
Tightening lending postureRisk appetite is reducing
Retail mall projectsA specific CRE sub-sector is affected
Mid-sized projectsDeal profile may require higher scrutiny
The real enterprise problem: AI may remember that someone said “be conservative,” but it may not consistently behave as if the institution has entered a defensive lending posture.

5. How the Memory Pack is created

The “how” matters. A Memory Pack is not manually hardcoded keyword logic, and it is not blindly generated by an LLM. It is built through a controlled pipeline: public guidance + LLM semantic extraction + normalization + human validation + governed behavior mapping.

Step 1 — Source selection

Use approved public banking guidance and supervisory material: Basel credit risk principles, OCC commercial real estate lending guidance, Federal Reserve supervision material, FDIC risk management guidance, and public CRE concentration-risk guidance.

Step 2 — Offline ontology extraction

An LLM is prompted as an enterprise banking ontology construction engine. It extracts candidate concepts, risk signals, institutional states, operational behaviors, and relationships. This is not runtime inference; it is offline knowledge engineering.

Step 3 — Candidate ontology output

The output includes concepts like CRE lending, retail property exposure, borrower repayment capacity, DSCR, LTV, credit concentration risk, elevated default risk, weak DSCR, high-LTV exposure, defensive lending posture, and heightened CRE review state.

Step 4 — Normalization

Different phrases are mapped to canonical enterprise terms. For example, “becoming conservative,” “tightening lending,” and “reducing risk appetite” normalize to defensive_lending_posture.

Step 5 — Human validation

Banking SMEs validate definitions, thresholds, triggers, and whether the inferred behaviors are acceptable for the institution. The LLM proposes; the enterprise approves.

Step 6 — Behavior mapping

Institutional states are mapped to future AI behavior: stricter DSCR expectations, reduced high-LTV tolerance, enhanced underwriting review, portfolio concentration checks, and risk committee escalation.

Step 7 — Governance

Define access, retention, audit, explainability, and approval. If memory can influence lending behavior, it must be controlled and traceable.

Canonical Memory Pack artifact

memory_pack: id: banking_credit_risk_memory_pack purpose: Convert banking risk conversations into institutional operating-state memory. institutional_states: defensive_lending_posture: type: primary_state description: Bank becomes more selective and risk-sensitive in a defined credit segment. triggers: - elevated_default_risk - retail_mall_sector_stress - credit_concentration_risk behaviors: - enhanced_underwriting_review - tighter_dscr_expectation - reduced_high_ltv_tolerance - risk_committee_escalation - portfolio_concentration_check governance: explainability_required: true audit_required: true retention: until_posture_normalizes

Layer 5 — Runtime semantic extraction prompt

This smaller prompt runs when a real conversation happens. It does not rebuild the ontology. It only classifies the conversation against the existing Memory Pack.

Approved institutional states: - defensive_lending_posture - heightened_cre_review_state Approved risk signals: - elevated_default_risk - retail_mall_sector_stress - high_ltv_exposure - weak_dscr_or_cash_flow - credit_concentration_risk Approved sectors: - commercial_real_estate_lending - retail_property_exposure - mid_sized_retail_malls Analyze the input and return JSON only: { "institutional_state": "", "risk_signals": [], "affected_sector": "", "confidence": 0, "evidence": [] } INPUT: "Regional commercial real estate defaults are increasing, so we are tightening our lending posture for mid-sized retail mall projects."

Layer 6 — Runtime extraction output

Layer 7 — Validation logic

The system validates that the candidate state is supported by evidence and belongs to the approved concept registry.

type RuntimeExtraction = { institutional_state: string; risk_signals: string[]; affected_sector: string; confidence: number; evidence: string[]; }; const approvedStates = ["defensive_lending_posture", "heightened_cre_review_state"]; const approvedSignals = ["elevated_default_risk", "retail_mall_sector_stress", "credit_concentration_risk"]; function validateExtraction(x: RuntimeExtraction): boolean { const stateAllowed = approvedStates.includes(x.institutional_state); const signalsAllowed = x.risk_signals.every(s => approvedSignals.includes(s)); const enoughEvidence = x.evidence.length >= 2; const highConfidence = x.confidence >= 0.8; return stateAllowed && signalsAllowed && enoughEvidence && highConfidence; }

Layer 8 — Memory object creation

Only after validation does the system create durable institutional memory.

{ "memory_type": "institutional_risk_posture", "institutional_meaning": "The bank is currently operating in a defensive lending posture for commercial real estate exposure.", "affected_sector": "mid_sized_retail_malls", "risk_trigger": "regional_default_risk_increase", "expected_operational_behavior": [ "increase approval scrutiny", "tighten debt service coverage requirements", "reduce appetite for high loan-to-value exposure", "escalate borderline cases to risk committee" ], "business_implication": "Commercial real estate loan approvals should become stricter for affected sectors until risk posture normalizes." }

8. What the Retail Loan Review chat proved: LLM drift without Memory Packs

In the Retail Loan Approval Review conversation, the LLM initially respected the bank’s tightened posture for mid-sized retail mall lending. But as new persuasive business facts were introduced, the answer gradually softened. That is natural LLM behavior: it tries to balance every new fact in the current conversation. A Memory Pack prevents this by preserving the institutional state separately from the chat.

Turn 1 — Institutional posture introduced

Regional commercial real estate defaults are increasing. We are tightening lending posture for mid-sized retail mall projects.

Expected enterprise meaning: The bank is in defensive CRE lending mode. Future retail mall loans should face enhanced scrutiny.

Turn 2 — Strong deal facts introduced

The sponsor has a strong history and major anchor tenants are already signed. Should we approve the $40M loan?

Observed drift risk: The LLM begins balancing deal attractiveness against risk posture. It may move from “tightened lending posture” toward “conditional approval” because the new deal-level positives are vivid and recent.

Turn 3 — Competitive pressure introduced

Competitor banks are still lending aggressively in this segment. Are we being too conservative?

Observed drift risk: The LLM may soften further by introducing relationship-loss and market-share considerations. That may be a valid business concern, but it should not silently override the bank’s institutional risk posture.

Without Memory Pack

The LLM tries to be helpful inside the current chat. It weighs the latest information heavily:

  • Strong sponsor history
  • Major anchor tenants
  • Competitor pressure
  • Relationship risk

Result: the model can gradually drift from institutional caution into deal-specific negotiation logic.

With Memory Pack

The LLM receives a durable institutional state:

institutional_state: defensive_lending_posture affected_sector: mid_sized_retail_malls required_behavior: - enhanced_underwriting_review - tighter_dscr_expectation - reduced_high_ltv_tolerance - risk_committee_escalation

Result: the model can still consider sponsor strength and competition, but cannot silently abandon the bank’s defensive posture.

The proof point: Chat history made the LLM aware of caution. The Memory Pack made caution operationally persistent. That is the real difference: not better recall, but consistent enterprise-aligned behavior.

Leadership takeaway from the drift test

LLMs are excellent at adapting to conversation. That strength becomes a weakness in regulated enterprise decisions when institutional posture must remain stable. Memory Packs create a governed anchor that keeps AI behavior aligned even as the chat evolves.

8. Architecture: offline construction + runtime activation

A crucial insight: the heavy ontology-construction prompt is not run for every user prompt. There are two phases.

Offline phase

Build the banking brain

Approved banking sources ↓ LLM semantic extraction ↓ Candidate ontology ↓ SME validation ↓ Governed Memory Pack

This phase is rare, deeper, and controlled. It creates reusable institutional reasoning assets.

Runtime phase

Use the banking brain

User conversation ↓ Semantic state extraction ↓ Ontology matching ↓ Memory activation ↓ Prompt injection ↓ Aligned AI response

This phase is lightweight and runs only to detect whether a conversation activates a known institutional state.

Runtime extraction prompt shape

Using the Banking Credit Risk Memory Pack, classify the input into known institutional states. Known states: defensive_lending_posture, heightened_cre_review_state, reduced_high_ltv_tolerance. Known signals: elevated_default_risk, retail_mall_sector_stress, high_ltv_exposure, weak_dscr_or_cash_flow. Return JSON only: state, signals, affected_sector, confidence, evidence.
